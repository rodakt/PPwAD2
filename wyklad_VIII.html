Rekurencja

Funkcj nazywamy rekurencyjn, gdy wywouje sam siebie. Wywoanie to moe by bezporednie lub porednie. Implementacja rekurencji polega czsto na obserwacji, e rozwizanie problemu dla pewnych szczeglnych argumentw jest oczywiste, a dla argumentw oglnych prosto wyraa si przez rozwizanie tego samego problemu dla argumentw "prostszych", znajdujcych si w jakim sensie "bliej" przypadku oczywistego.

Przykad. Chcemy napisa funkcj suma_naturalnych(n) zwracajc sum kolejnych liczb naturalnych od 1 do n:

1 + 2 + ... + n

Przypadek szczeglny to argument n rwny 0. Wtedy mamy sum po zbiorze pustym, czyli 0. Nic nie trzeba liczy, wynik jest oczywisty. Dla n przekraczajcych 0 mamy przypadek oglny, o ktrym mylimy tak: gdyby suma dla argumentu n - 1 bya ju obliczona, to wynikiem byaby warto tej sumy powikszona o n. Inaczej mwic:

  • suma_naturalnych(0) zwraca 0,
  • suma_naturalnych(n) zwraca n + suma_naturalnych(n - 1) dla n > 0.

Implementacja rekurencyjna funkcji suma_naturalnych():

>>> def suma_naturalnych(n):
...     '''Zwraca 1 + 2 + ... + n'''
...     if n == 0:
...         return 0
...     return n + suma_naturalnych(n - 1)
>>> suma_naturalnych(5)
15

Przebieg wywoania suma_naturalnych(5):

suma_naturalnych(5)
5 + suma_naturalnych(4)
5 + (4 + suma_naturalnych(3))
5 + (4 + (3 + suma_naturalnych(2)))
5 + (4 + (3 + (2 + suma_naturalnych(1))))
5 + (4 + (3 + (2 + (1 + suma_naturalnych(0)))))
5 + (4 + (3 + (2 + (1 + 0))))
5 + (4 + (3 + (2 + 1)))
5 + (4 + (3 + 3))
5 + (4 + 6)
5 + 10
15

Zauwa, e ten "wybrzuszajcy" si acuch operacji musi by zapamitywany. Powiemy, e funkcja suma_naturalnych() jest rekurencyjna oraz, e jej wywoanie generuje proces rekurencyjny, czyli buduje acuch odoonych operacji, ktre musz by zapamitywane; im duszy proces, tym wicej operacji zostaje odoonych.

W jzyku Python istnieje limit na liczb odoonych operacji. Przekroczenie tego limitu skutkuje wyjtkiem RecursionError:

>>> import sys
>>> sys.getrecursionlimit()
3000
>>> suma_naturalnych(3000)
Traceback (most recent call last):
  ...
RecursionError: maximum recursion depth exceeded in comparison
>>> sys.setrecursionlimit(4000)
>>> suma_naturalnych(3000)
4501500

Nasza funkcja suma_naturalnych() nie ma zatem zbyt wielu zalet w porwnaniu z tradycyjn implementacj za pomoc ptli. Nie do, e zajmuje pami -- tym wicej im wiksze n -- to jeszcze do tego nie jest w stanie zwrci wyniku dla nieduego przecie n rwnego 3000.

Popatrzmy chwil na nasze sumowanie zakodowane przy pomocy ptli while:

>>> def suma_naturalnych(n):
...     '''Zwraca 1 + 2 + ... + n'''
...     k, suma = 0, 0
...     while k < n:
...         k += 1
...         suma += k
...     return suma
>>> suma_naturalnych(5)
15

W ptli obsugujemy dwie zmienne: kroczc sum suma i dodawan warto k. Tak samo bdzie, gdybymy wykorzystali ptl for, tyle e wtedy ptla sama zajmie si aktualizacj wartoci k:

>>> def suma_naturalnych(n):
...     '''Zwraca 1 + 2 + ... + n'''
...     suma = 0
...     for k in range(1, n + 1):
...         suma += k
...     return suma
>>> suma_naturalnych(5)
15

Okazuje si, e rekurencja potrafi odwzorowa zwyk ptl while czy for. Zmienne (by moe liczne) przetwarzane w ptli przekazuje si kolejnym wywoaniom rekurencyjnym za pomoc argumentw. Oto ilustracja dla funkcji suma_naturalnych():

>>> def sumuj(n, suma):
...     '''Zwraca suma + (1 + 2 + 3 + ... + n)'''
...     if n == 0:
...         return suma
...     return sumuj(n - 1, n + suma)
>>> def suma_naturalnych(n):
...     '''Zwraca 1 + 2 + ... + n'''
...     return sumuj(n, 0)
>>> suma_naturalnych(5)
15

W powyszej implementacji rekurencyjna jest funkcja sumuj(). Rozpiszmy przykadowy acuch wywoa:

suma_naturalnych(5)
sumuj(5, 0)
sumuj(4, 5)
sumuj(3, 9)
sumuj(2, 12)
sumuj(1, 14)
sumuj(0, 15)
15

Tym razem nie ma acucha odoonych operacji, proces si nie "wybrzusza". Funkcja sumuj() jest rekurencyjna, ale implementuje proces iteracyjny; operacje dodawania wykonywane od razu i zapamitywane w dodatkowej zmiennej suma. Z tego punktu widzenia rekurencyjno sumuj() jest jedynie wasnoci skadniow, samo jednak wywoanie jest rwnowane zwykej ptli.

O funkcji rekurencyjnej powiemy, e jest rekurencj ogonow, jeli warto zwracana przez ostatnie wywoanie rekurencyjne jest finaln wartoci zwracan przez funkcj. W powyszym przykadzie ostatnie wywoanie to sumuj(0, 15), ktre zwraca 15 co jest ostatecznym wynikiem. Tymczasem tutaj

suma_naturalnych(5)
5 + suma_naturalnych(4)
5 + (4 + suma_naturalnych(3))
5 + (4 + (3 + suma_naturalnych(2)))
5 + (4 + (3 + (2 + suma_naturalnych(1))))
5 + (4 + (3 + (2 + (1 + suma_naturalnych(0)))))
5 + (4 + (3 + (2 + (1 + 0))))
5 + (4 + (3 + (2 + 1)))
5 + (4 + (3 + 3))
5 + (4 + 6)
5 + 10
15

ostanim wywoaniem jest suma_naturalnych(0), co oczywicie nie jest finalnym wynikiem. eby ten wynik uzyska rodowisko uruchmieniowe musi teraz wrci przez wszystkie odoone na stosie wywoania a do pierwszego, ktre zwrci warto i zakoczy dziaanie funkcji. Dlatego sumuj() jest rekurencj ogonow, ale nie jest ni suma_naturalnych() zaprezentowana na grze strony.

Jeeli rodowisko "wie", e wywoywana rekurencja jest ogonowa, to wygenerowane po drodze wywoania moe porzuca, gdy i tak nie wpyn na ostateczny wynik. Wywoanie sumuj(4, 5) nie musi by pamitane, jeli dziaa ju kolejne wywoanie sumuj(3, 9). Po zakoczeniu dziaanie sumuj(3, 9) uruchamiane jest sumuj(2, 12), zatem rodowisko lokalne utworzone przez sumuj(3, 9) moe zosta bez szkody zapomniane, itd.

Mwimy, e jzyk programowania posiada optymalizacj ogonow (tail call optimization), jeli ma zdolno zapominania zbdnych odwoa rekurencyjnych w rekurencji ogonowej. Jzyk Python takiej zdolnoci nie posiada. Mimo, e tworzenie acucha odoonych operacji nie jest konieczne dla wywoania sumuj(), to i tak dla duych n to wywoanie wysypuje si wyjtkiem RecursionError:

>>> sumuj(3000, 1)
Traceback (most recent call last):
  ...
RecursionError: maximum recursion depth exceeded in comparison

Sprawne korzystanie z rekurencji wymaga praktyki. W wielu jzykach programowania brak optymalizacji ogonowej, co powoduje, e rozwizanie rekurencyjne jest czsto mniej wydajne ni rwnowane rozwizanie wykorzystujce ptle. Ponadto, niektre jzyki programowania bardziej wspieraj idiom rekurencji, inne mniej. Korzystanie z ptli w Pythonie, zwaszcza z ptli for po kolekcjach, jest bardzo naturalne. Nie ma wielkiego sensu w zastpowaniu takiej ptli wymyln rekurencj ogonow, ktra i tak nie zostanie zoptymalizowana. Z drugiej strony, w jzykach takich jak Lisp, jego dialekt Scheme czy Haskell nie ma w ogle instrukcji sterujcych odpowiadajcych zwykym ptlom while czy for z jzykw imperatywnych. Zachowanie ptli implementowane jest rekurencj (oczywicie ze wspieran optymalizacj ogonow) lub wykonywane jest niejawnie przez wywoania funkcyjne. W tych jzykach stoswanie rekurencji jest bardzo naturalne.

Rekurencja jest oglniejsza od ptli, jednak rozwizania wykorzystujce ptle uchodz za atwiejsze do zrozumienia. Zasada ta ma wiele wyjtkw. Zdarza si, e rozwizanie rekurencyjne jest znacznie prostsze do znalezienia ni jakiekowlwiek rozwizanie za pomoc ptli. Przykadu dostarcza problem liczby podziaw liczby naturalnej, o ktrym moesz poczyta tutaj w Composing Programs.

Bardzo wana metodyka projektowania algorytmw dziel i zwyciaj dostarcza wielu przykadw problemw z wydajnymi rozwizaniami rekurencyjnymi. Czoowe przykady to wyszukiwanie binarne, sortowanie szybkie i sortowanie przez scalanie.

Rekurencja w cigu Fibonacciego

Definicja i analiza wydajnoci

Cig Fibonacciego matematycy definiuj nastpujco: $$F_0 := 0,\quad F_1 := 1$$ i dla $n\geqslant 2,$ $$F_n := F_{n-2} + F_{n-1}.$$

W wielu jzykach programowania definicja ta daje si wprost przeoy na definicj funkcji obliczajcej wyraz o indeksie n:

>>> def fib(n):
...     '''Zwraca wyraz F_n, n >= 0 cigu Fibonacciego.'''
...     if n == 0 or n == 1:
...         return n
...     return fib(n - 1) + fib(n - 2)

Sprawdmy jak skaluje si czas wykonania fib(n) wraz ze wzrostem n. Czas bdziemy mierzy funkcj perf_counter(). Zaniedbujemy mae n, gdy dla nich czasy bd bardzo krtkie, a przez to jednorazowy pomiar naraony na due niedokadnoci:

In[1]:
%%time
from time import perf_counter

def fib(n):
    if n < 2:
        return n
    else:
        return fib(n - 1) + fib(n - 2)

N, T = [], []
n_start, n_stop = 10, 37
# Rozmiary prbki dla poszczeglnych n; 50 dla n < 30, 5 dla n >= 30
# Czas wykonania jest uredniany dla kadej prbki
rozmiary_prby = [100 if n < 30 else 5 for n in range(n_start, n_stop + 1)]

for n, rozmiar_prbki in zip(range(n_start, n_stop + 1), rozmiary_prby):
    czasy = []
    for _ in range(rozmiar_prbki):
        start = perf_counter()
        w = fib(n)
        stop = perf_counter()
        czas = stop - start
        czasy.append(czas)
    N.append(n)
    T.append(sum(czasy) / rozmiar_prbki)
    print(f'{n:<3}: {w:<10}| Czas: {czas}')

Odchylenia od redniej mog by spowodowane prac systemu operacyjnego. Wykres ten bdzie ulega zmianie po kadorazowym przeliczeniu list N i T. Jeli przyjmiemy, e ilorazy czasw wykonania s stae, to sam cig czasw wykonania powinien by dobrze przybliany przez cig geometryczny

$$T_n = a \cdot q^n,$$

dla pewnych staych $a$ i $q$. Logarytmujc obie strony rwnania, dostajemy

$$\log T_n = \log a + n \cdot \log q.$$

Teraz wspczynniki $\log a$ i $\log q$ moemy obliczy, korzystajc z regresji liniowej:

Napiszmy zatem dekorator, ktry doda atrybut liczba_wywoa do zmodyfikowanej funkcji. Atrybut ten ustawiamy na zero, przy kadorazowym wywoaniu funkcji jego warto podnosimy o jeden:

In[14]:
from functools import wraps

def licz_wywoania(f):
    '''Przechowuje liczb wywoa f() w atrybucie f.liczba_wywoa.'''
    @wraps(f)
    def licznik(*args, **kwargs):
        wynik = f(*args, **kwargs)
        licznik.liczba_wywoa += 1
        return wynik
    licznik.liczba_wywoa = 0
    return licznik

def suma_naturalnych(n):

    if n == 0:

    return n + suma_naturalnych(n - 1)

Ratowanie definicji rekurencyjnej

W bezporedniej implementacji definicji rekurencyjnej cigu Fibonacciego problemem jest to, e na jednej i tej samej liczbie k funkcja fib() wywoywana jest wielokrotnie.

Z kopotem tym poradzimy sobie zapamitujc wyniki porednie:

In[29]:
pami = {}

def fib_z_pamici(n):
    '''Zwraca wyraz F_n, n >= 0 cigu Fibonacciego.'''
    if n == 0 or n == 1:
        return n
    
    if n not in pami:
        pami[n] = fib_z_pamici(n - 2) + fib_z_pamici(n - 1)
        
    return pami[n]

Dekorator zapamitujcy

Podany wyej chwyt z pamici mona przeoy na dekorator zapamitujcy stosowalny do funkcji jednej zmiennej.

In[37]:
def zapamituj(f):
    pami = {}
    @wraps(f)
    def wrapper(x):
    
        if x not in pami:
            pami[x] = f(x)
        
        return pami[x]
    
    return wrapper

Moemy go zastosowa do poprzedniej funkcji fib(). Dziki uyciu dekoratora zachowujemy czytelno definicji rekurencyjnej.

In[38]:
@zapamituj
def fib(n):
    '''Zwraca wyraz F_n, n >= 0 cigu Fibonacciego.'''
    if n == 0 or n == 1:
        return n
    
    return fib(n - 2) + fib(n - 1)

functools.lru_cache

Dekorator lru_cache() z moduu functools implementuje memoizacj (czyli zapamitywanie porednich wynikw):

In[43]:
from functools import lru_cache

@lru_cache()
def fib(n):
    '''Zwraca wyraz F_n, n >= 0 cigu Fibonacciego.'''

    if n == 0 or n == 1:
        return n
    
    return fib(n - 1) + fib(n - 2)

lru_cache() rwnie przyjmuje parametry. Dlatego naley go pisa z nawiasami okrgymi -- nawet wtedy, gdy adnych argumentw nie wprowadzamy.
